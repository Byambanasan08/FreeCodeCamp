Algorithm гэж юу вэ, мөн Big O Notation хэрхэн ажилладаг вэ?

Таны төхөөрөмж дээр ажиллаж буй компьютерийн бүх програмууд тодорхой дарааллаар гүйцэтгэгддэг зааврын (instructions) багцтай байдаг.

Эдгээр зааврууд нь тодорхой нэг даалгаврыг гүйцэтгэхэд зориулагдсан байдаг.

Тэр даалгавар нь тоонуудын жагсаалтыг эрэмбэлэх (sorting), зураг өөрчлөх (modifying an image), inventory хянах, эсвэл бүр таны дуртай video game-ийг ажиллуулах ч байж болно.

Энд алгоритм (algorithm) орж ирдэг. Алгоритм гэдэг нь асуудлыг шийдэх эсвэл даалгаврыг гүйцэтгэхэд зориулсан тодорхой, хоёрдмол утгагүй (unambiguous) заавруудын багц юм.

Та алгоритмыг “жор” (recipes) гэж төсөөлж болно. Та хоол хийх үед жор нь хэрэгтэй бүх орцуудыг жагсааж, хоолыг хэрхэн бэлтгэхийг алхам алхмаар зааж өгдөг.

Үүнтэй адил алгоритм нь компьютерт яг юу хийх ёстой болон яаж хийхийг зааж өгдөг “жор” юм.

Алгоритм нь хоёр гол шинж чанартай:

```
Энэ нь төгсгөлгүй (indefinitely) үргэлжилж болохгүй. Энэ нь тодорхой хязгаарлагдмал (finite) алхмын тоонд дуусах ёстой.

Алхам бүр нь тодорхой, хоёрдмол утгагүй (precise and unambiguous) байх ёстой.
```

Алгоритм нь 0, 1, эсвэл олон оролт (inputs)-той байж болох ба нэг эсвэл олон гаралт (outputs) үүсгэж болно.

Алгоритмын алхмууд нь ямар нэг програмчлалын хэлээс хамааралгүй байдаг.

Гэхдээ тэдгээрийг компьютерт ажиллуулахын тулд та Python эсвэл JavaScript зэрэг програмчлалын хэл дээр хэрэгжүүлэх (implement) хэрэгтэй.

Хэрэв алгоритм зөв бол, ямар ч хүчинтэй (valid) оролтод (input) зориулсан гаралт нь хүлээгдэж буй (expected) гаралттай таарах ёстой.

Зөв байхаас гадна алгоритм нь үр ашигтай (efficient) байх ёстой.

Алгоритмын үр ашгийг (efficiency) тэдгээрийг ажиллуулахад хэр хугацаа шаардагдах болон даалгаврыг гүйцэтгэхэд санах ойд (memory) хэр их зай шаардлагатайгаар хэмжиж болно.

Алгоритмын үр ашгийг мэдэх нь маш чухал, учир нь оролтын хэмжээ (input size) нэмэгдэхэд алгоритм хэр сайн ажиллахыг ойлгоход тусалдаг.

Жишээлбэл, 15 ширхэг бүхэл тоог эрэмбэлэх нь 1 сая бүхэл тоог эрэмбэлэхтэй адил биш.

Процессийн хэмжээ болон төвөгшил (complexity) нэмэгдэх тусам, хэрэв алгоритм хангалттай үр ашигтай биш бол та маш удаан ажиллах програмтай болох эсвэл бүр бүх системийг crash болгож болзошгүй.

Тиймээс хамгийн үр ашигтай алгоритмыг хөгжүүлэх болон сонгох нь маш чухал юм.

Энд Big O notation маш чухал болдог.

Big O notation нь оролтын хэмжээ нэмэгдэхэд алгоритмын хамгийн муу нөхцөл (worst-case performance) буюу өсөлтийн хурдыг (growth rate) тодорхойлдог.

Алгоритмын өсөлтийн хурд гэдэг нь оролтын хэмжээ нэмэгдэхэд шаардлагатай нөөцүүд (resources) хэрхэн нэмэгдэж байгааг илэрхийлдэг.

Big O notation нь хамгийн муу нөхцөлд анхаардаг, учир нь энэ нь ямар ч оролт байсан алгоритм хэр үр ашигтай ажиллахыг ойлгоход чухал.

Манай өмнөх эрэмбэлэх жишээ рүү буцъя, 1 сая бүхэл тоог эрэмбэлэх нь 15 бүхэл тоог эрэмбэлэхээс илүү их хугацаа болон нөөц шаардах нь ойлгомжтой.

Гэхдээ хэр их илүү вэ?

Энэ нь таны сонгосон алгоритмаас хамаарна.

Big O notation нь алгоритмын үр ашгийг тодорхойлох яг тоон утга өгөхгүй, харин алгоритм гүйцэтгэж буй үйлдлийн (operations) тоонд үндэслэн оролтын хэмжээ нэмэгдэхэд хэрхэн өсөж байгааг ойлгуулахад тусалдаг.

Big O notation-д ихэвчлэн оролтын хэмжээг n үсгээр тэмдэглэдэг. Жишээлбэл, хэрэв оролт нь жагсаалт (list) бол n нь тэр жагсаалтын элементүүдийн тоог илэрхийлнэ.

Алгоритмын хугацааны төвөгшлийг (time complexity) тодорхойлоход тогтмол үржвэрүүд (constant factors) болон доод зэрэглэлийн гишүүд (lower-order terms)-ийг тооцохгүй. Учир нь n-ийн хэмжээ өсөх тусам эдгээр жижиг гишүүдийн нөлөө улам багассаар байдаг.

Алгоритмын нийт зан төлөвийг (overall behavior) тодорхойлох гол зүйл нь хамгийн өндөр зэрэглэлийн (highest order) n-тэй гишүүн байдаг.

Жишээлбэл, хэрэв алгоритм 7n + 20 үйлдэл гүйцэтгэдэг бол n өсөх тусам тогтмол 20-ын нөлөө багасна. 7n гишүүн давамгайлж, алгоритмын нийт зан төлөв болон үр ашгийг тодорхойлно.

Өөр нэг жишээ нь, алгоритм 20n² + 15n + 7 үйлдэл гүйцэтгэдэг байж болно. n өсөх тусам 20n² гишүүн давамгайлах тул энэ алгоритм квадрат хугацааны төвөгшилтэй (quadratic time complexity) гэж үзнэ, учир нь давамгайлж буй гишүүн нь n² байна.

Квадрат хугацааны төвөгшил (quadratic time complexity) нь алгоритмын ертөнцөд байдаг олон төрлийн хугацааны төвөгшлүүдийн нэг юм.

Одоо хамгийн түгээмэл зарим төрлүүдийг сурцгаая.

O(1)-ийг “Constant Time Complexity” гэж нэрлэдэг. Хэрэв алгоритм тогтмол хугацааны төвөгшилтэй (constant time complexity) бол оролтын хэмжээ ямар байхаас үл хамааран ижил хугацаанд ажиллана.

Жишээлбэл, тоо тэгш (even) эсвэл сондгой (odd) эсэхийг шалгах нь тухайн тооноос үл хамааран үргэлж ижил хугацаа шаарддаг.

```js id="k7m3vp"
function checkEvenOrOdd(number) {
  if (number % 2 === 0) {
    return "Even";
  } else {
    return "Odd";
  }
}
```

O(log n)-ийг “Logarithmic Time Complexity” гэж нэрлэдэг. Энэ нь оролтын хэмжээ нэмэгдэхэд алгоритмын шаардлагатай хугацаа удаанаар (slowly) өсдөг гэсэн үг. Энэ нь асуудлын хэмжээг тогтмол хувиар (constant fraction) давтан багасгадаг асуудлуудад түгээмэл тохиолддог.

Жишээлбэл, Binary Search нэртэй алдартай хайлтын алгоритм нь хамгийн муу нөхцөлд O(log n) хугацааны төвөгшилтэй байдаг. Учир нь энэ нь харьцуулалт бүр дээр үлдсэн элементүүдийн талыг хасдаг тул ерөнхийдөө илүү үр ашигтай байдаг.

O(n)-ийг “Linear Time Complexity” гэж нэрлэдэг. Энэ төрлийн алгоритмуудын ажиллах хугацаа нь оролтын хэмжээтэй шууд пропорциональ (proportionally) өсдөг.

Жишээлбэл, жагсаалтын (list) бүх элементүүдээр давтах for loop нь элементүүдийн тоо нэмэгдэх тусам илүү олон давталт гүйцэтгэнэ. Хэрэв жагсаалтын хэмжээ хоёр дахин нэмэгдвэл, үйлдлийн тоо ойролцоогоор хоёр дахин нэмэгдэнэ.

```js id="p2v9qt"
for (const grade of grades) { // grades нь array
  console.log(grade);
}
```

O(n log n)-ийг “Log-Linear Time Complexity” гэж нэрлэдэг. Энэ нь Merge Sort болон Quick Sort зэрэг үр ашигтай эрэмбэлэх (sorting) алгоритмуудын түгээмэл хугацааны төвөгшил юм.

O(n²)-ийг “Quadratic Time Complexity” гэж нэрлэдэг. Эдгээр алгоритмуудын ажиллах хугацаа оролтын хэмжээтэй квадрат (quadratically) харьцаагаар өсдөг бөгөөд бодит амьдралын (real-world) асуудлуудад ерөнхийдөө үр ашиг муутай байдаг.

Nested loop-ууд нь квадрат хугацааны төвөгшлийн нийтлэг жишээ юм. Дотоод (inner) loop нь гаднах (outer) loop-ийн n удаагийн давталт бүрд n удаа давтагддаг тул нийтдээ n квадрат (n²) давталт үүснэ.

```js id="t5m8nx"
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {
    console.log("Hello, World!");
  }
}
```

Бусад хугацааны төвөгшлүүдэд “Exponential Time Complexity” (O(2^n)) болон “Factorial Time Complexity” (O(n!)) орно. Эдгээр нь бодит амьдралын нөхцөлд үр ашиг муутай байдаг.

Энэ график дээр та хамгийн түгээмэл хугацааны төвөгшлүүдийг илэрхийлдэг математик функцүүдийн өсөлтийг харьцуулж болно. x тэнхлэгийг (horizontal) оролтын хэмжээ гэж, харин y тэнхлэгийг (vertical) алгоритмын ажиллах хугацаа гэж бодоорой.

Та Quadratic Time Complexity (O(n²)) (шар) нь бусдаасаа хамаагүй хурдан өсдөгийг, харин Constant Time Complexity (O(1)) (улаан) нь оролтын хэмжээ нэмэгдсэн ч тогтмол хэвээр байгааг харж болно.

Сайн байна. Одоог хүртэл та Big O notation-ийг хугацааны шаардлагын (time requirements) үүднээс сурсан, харин энэ тэмдэглэгээг (notation) санах ойн (space requirements) үүднээс мөн ашиглаж болно.

Энэ context-д, алгоритмд шаардлагатай санах ойн хэмжээ оролтын хэмжээ өсөх тусам хэрхэн нэмэгдэж байгааг тайлбарладаг.

“Constant Space Complexity” O(1)-тэй алгоритмууд нь оролтын хэмжээ нэмэгдсэн ч үргэлж тогтмол хэмжээний санах ой шаарддаг.

Үүний жишээ нь санах ойд цөөн хэдэн хувьсагч (variables) үүсгэж хадгалдаг алгоритм байж болно.

Харин “Linear Space Complexity” O(n)-тэй алгоритмуудын санах ойн шаардлага нь оролтын хэмжээ нэмэгдэх тусам пропорциональ (proportionally) өсдөг.

Үүний жишээ нь урт нь n байх жагсаалтын (list) хуулбарыг үүсгэж хадгалдаг алгоритм юм.

Мөн “Quadratic Space Complexity” O(n²)-тэй алгоритмуудын санах ойн шаардлага нь оролтын хэмжээ нэмэгдэх тусам квадрат (quadratically) өсдөг.

Үүний жишээ нь оролтын хэмжээнээс хамаарч бүх боломжит хослолуудыг хадгалдаг 2 хэмжээст матриц (2D matrix) үүсгэх явдал юм.

Алгоритмууд нь компьютерийн програмуудын суурь (building-blocks) болдог бол Big O notation нь тэдгээрийн үр ашгийг (efficiency) шинжлэх хүчирхэг framework бөгөөд оролтын хэмжээ өсөхөд хамгийн муу нөхцөлд (worst-case scenario) хугацаа болон санах ойн шаардлага хэрхэн өсөхийг тодорхойлдог. Тэдгээрийн үр ашгийг ойлгох нь бодит амьдралын нөхцөлд (real-world scenarios) үр ашигтай ажиллах програм хангамж (software) хөгжүүлэхэд маш чухал юм.
