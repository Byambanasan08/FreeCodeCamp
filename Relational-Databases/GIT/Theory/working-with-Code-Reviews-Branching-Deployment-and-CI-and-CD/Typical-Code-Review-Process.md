# Төслүүдийн хувьд ердийн code review процесс ямар байдаг вэ?

Өмнөх хичээл дээр та өөрийн repository руу pull request хэрхэн хийхийг сурсан. Мөн тэр pull request-ийг хэрхэн merge хийхийг сурсан. Гэхдээ та үүнийг зөвхөн repository таных байсан учраас хийж чадсан — өөрийн repository дээр та maintainer эрхтэй байдаг бөгөөд энэ нь pull request-уудыг merge хийх зөвшөөрөлтэй гэсэн үг.

Харин та бусдын open source төсөлд хувь нэмэр оруулах үед ихэнхдээ ийм зөвшөөрөл байхгүй. Үүний оронд maintainers code review процессоор таны кодыг шалгаж, таны хувь нэмрийг хүлээн зөвшөөрөхийг хүлээх хэрэгтэй болно. Тэд дараа нь merge хийх ажлыг гүйцэтгэнэ.

Гэхдээ code review процесс яг ямар байдаг вэ?

Maintainers-ийн хэрэгжүүлэх процесс нь хүн бүрээс хамаарч өөр өөр байж болох бөгөөд төслөөс хамаарсан тусгай арга барил ч байж болно. Гэхдээ ерөнхийдөө code review нь ихэвчлэн гурван алхамтай байдаг.

Эхний алхам нь CI (continuous integrations)-ийг шалгаж, амжилттай (pass) болсон эсэхийг баталгаажуулах явдал юм. Та энэ талаар дараагийн хичээлүүдэд илүү дэлгэрэнгүй судлах болно, одоохондоо үүнийг код нь style guideline-уудыг хангаж байгаа эсэх болон test suite-ийг давж байгаа эсэхийг шалгадаг хяналт (checks) гэж ойлгож болно.

Хэрвээ CI амжилтгүй бол maintainers ихэвчлэн pull request-ийг шинэчилж, алдаануудыг засахыг хүснэ. Эдгээр алдаанууд нь юу буруу болсон талаар харах боломжтой ямар нэг тайлан (report)-тай байдаг, мөн maintainers заримдаа асуудлыг хэрхэн шийдэх талаар илүү нарийн заавар өгч болно.

Checks амжилттай болсны дараа maintainers таны бодит кодын өөрчлөлтүүдийг шалгана. Тэд diff харагдацаар (view) дамжуулан та юу өөрчилсөн, нэмсэн, устгасныг харж шинжилж чадна. Тэд яагаад тодорхой арга барил сонгосныг асууж магадгүй тул та тайлбар өгөхдөө бодлоготой, нягт нямбай байх нь зүйтэй.

Мөн тэд нэмэлт өөрчлөлт хийхийг хүсэж болно. Энэ нь ямар байдлаар явагддаг талаар бид удахгүй ярилцана.

Сүүлийн алхам нь ихэвчлэн таны өөрчлөлтүүдийг local орчинд татаж авах байдаг. Maintainers дараа нь таны өөрчлөлттэйгээр application-ийг ажиллуулж, функциональ байдлыг шалгаж, ямар нэг зүйл эвдэрсэн эсвэл гэнэтийн байдлаар ажиллаж байгаа эсэхийг баталгаажуулна.

Хэрвээ бүх зүйл зөв харагдвал maintainer таны pull request-ийг approve хийнэ — заримдаа “LGTM” гэсэн comment-оор, энэ нь “looks good to me” гэсэн утгатай. Pull request тань approve хийгдсэний дараа maintainers өөрчлөлтүүдийг оруулахад бэлэн болсон үед merge хийнэ. Зарим төслүүд нэгээс олон approval шаарддаг — жишээ нь freeCodeCamp хоёр approval шаарддаг.

Харин таны pull request дээр ямар нэг зүйл буруу байвал яах вэ? Maintainers review хийж, өөрчлөлт шаардах боломжтой. Эдгээр шаардлагууд хэд хэдэн хэлбэртэй байж болно.

Заримдаа maintainer зүгээр л “comment” үлдээж, ямар нэг зүйлийг өөрчлөх эсвэл шинэчлэхийг санал болгож болно, гэхдээ үүнийг хэрхэн хэрэгжүүлэх тал дээр илүү уян хатан (flexible) байдлыг үлдээдэг.

Эсвэл maintainers албан ёсны “request changes” comment үлдээж болно. Ийм төрлийн comment ихэвчлэн pull request-ийг merge хийхийг хориглодог. Та шаардсан өөрчлөлтүүдийг хийж, дараа нь өөрчлөлт хүссэн тэр maintainer дахин шалгаж, дахин өөрчлөлт шаардах эсвэл approve хийх ёстой болно.

Maintainers тодорхой өөрчлөлт санал болгох үед GitHub-ийн UI ашиглан шууд санал (suggestion) оруулж чадна. Ийм шууд санал өгсөн тохиолдолд GitHub танд тухайн саналыг шууд өөрийн branch дээр commit хийх товч өгдөг. Гэхдээ ийм санал олон байвал files view рүү шилжих нь зүйтэй.

Тэр view-ээс та бүх саналуудаа (нэг нэгээр нь) “batch” болгон цуглуулж, нэг дор commit хийж чадна. Энэ нь маш олон санал байгаа үед маш ашигтай, учир нь тус бүрээр нь commit хийхэд олон commit үүсэж, CI шалгалтууд commit бүр дээр ажиллах шаардлагатай болж, PR-ийг батлах процессыг удаашруулдаг.

Хэрвээ шаардсан өөрчлөлтүүд томоохон байвал та өөрчлөлтүүдийг local орчинд хийж, PR-д ашигласан тэр branch дээр commit хийж, дараа нь push хийх нь зөв. GitHub автоматаар таны pull request-ийг шинэ commit-уудтай уялдуулан шинэчилнэ.

Мөн бид хамрах ёстой хэд хэдэн нэмэлт зүйлс байна.

Нэгдүгээрт, maintainers ч бас хүмүүс гэдгийг санаарай. Open source төслүүд ихэнхдээ maintainers-д бүрэн цагаар ажиллах хангалттай орлого оруулдаггүй тул тэд өөр ажилтай байдаг. Иймээс review процессын үед тэвчээртэй байх нь чухал — maintainers-ийг байн байн ping хийж, дахин шалгахыг шаардах эсвэл процессыг яаруулах оролдлого хийх хэрэггүй.

Хувь нэмэр оруулах нь сэтгэл хөдөлгөм байж болно! Гэхдээ процессыг бүү яаруул. Харин цаг гаргаж, нягт нямбай ажилла. Өөрчлөлтүүдээ local орчинд заавал тест хий — төслийн test suite байвал ажиллуул, development server-ийг асааж өөрчлөлтүүд чинь зөв ажиллаж байгаа эсэхийг гараар шалга, гэх мэт. Мөн ажиллаж буй төслийн contribution guidelines-ийг үргэлж дагаж мөрдөөрэй.

Хэрвээ энэ бүхнийг анхаарч, хамгийн сайн хүчин чармайлтаа гаргавал таны хувь нэмрүүд амжилттай явагдаж, code review ч эерэг approve-оор дуусах болно!
