# Git branching strategy (салбарын стратеги) боловсруулахад ямар сайн практикууд байдаг вэ?

Өмнөх хичээлүүдэд бид branch-уудын талаар бага зэрэг ярьсан. Та branch үүсгэхийг, яагаад main дээр шууд commit хийх ёсгүйг болон үүнтэй төстэй зүйлсийг сурсан. Гэхдээ яг хэзээ, яагаад branch үүсгэх ёстой вэ?

Юуны өмнө, яагаад зүгээр л олон commit-ийг main руу push хийж болохгүй вэ гэдгийг ярья. Таны default branch нь найдвартай, тогтвортой байх ёстой — өөрөөр хэлбэл, ямар ч үед хэн нэгэн таны main branch-ийг pull хийж аваад application-ийг build, run, test хийж чаддаг байх ёстой. Хэрвээ та дутуу дууссан feature-ууд эсвэл WIP (Work In Progress) commit-ууд push хийж байвал энэ нөхцөл үргэлж хангагдахгүй.

Нэмж хэлэхэд, олон feature-ууд дээр зэрэг ажиллах нь маш түгээмэл. Хэрвээ та олон feature-ийн дутуу ажлуудыг зэрэг main руу push хийж байвал асуудал гарсан үед debug хийх нь илүү хэцүү болно, учир нь өөрчлөлтүүд тусгаарлагдаагүй байна. Үүн дээр нэмээд context switching (өөр ажил руу шилжих) ч маш замбараагүй болно.

Тэгэхээр хэзээ branch үүсгэх вэ? Ерөнхий дүрэм бол тусгаарлагдсан ажлын нэгж бүрт (isolated unit of work) нэг branch үүсгэх явдал юм. Жишээ нь, та шинэ OAuth feature дээр ажиллаж байгаа бол тэр ажил тусгай branch дээр байх ёстой. Хэрвээ дараа нь password authentication дээр bug засах хэрэг гарвал тэр ажил өөрийн гэсэн branch дээр байх ёстой. User profile page-ийг дахин дизайн хийх хэрэгтэй юу? Тэр нь бас өөр branch.

Та branch-уудаа үргэлж main-ээс үүсгэх ёстой гэдгийг анхаараарай. Жишээ нь, та OAuth feature дээр ажиллаж байгаа branch дээр байгаа бол password authentication-ийн bug засах шинэ branch үүсгэхийн өмнө main руу checkout хийх ёстой. Эс бөгөөс OAuth branch-ээс шууд шинэ branch үүсгэвэл OAuth-ийн commit-ууд дагалдаж, history нь “булингартай” (muddied) болно.

Харин branch-уудаа яаж нэрлэх вэ? Энэ нь таны (эсвэл таны оролцож буй төслийн) шийдэх зүйл боловч дараах нийтлэг conventions байдаг:

scope/description — энд scope нь feat эсвэл fix гэх мэт байдаг, description нь branch-ийн өөрчлөлтийг товч тайлбарлана. Жишээ: feat/oauth-support.

scope/issue/description — өмнөхтэй адил боловч холбогдох issue-ийн дугаарыг нэмдэг. Жишээ: fix/25/password-auth.

username/scope/description — хэрвээ та fork ашиглахгүйгээр шууд анхны repository дээр branch ашиглаж ажилладаг төсөлд (ихэвчлэн private internal repository) оролцож байвал эхэнд нь username-ээ нэмэх нь ашигтай. Жишээ: naomi-lgbt/feat/profile-redesign.

Одоо branching strategy-тэй холбоотой хэд хэдэн нэмэлт зүйлийг авч үзье.

Урт хугацааны feature ажлууд (жишээ нь их хэмжээний developer-hour шаарддаг beta feature) дээр branch ашиглахын оронд feature flag ашиглан логикийг сонгож идэвхжүүлэх нь илүү тохиромжтой байж болно. Branch удаан хугацаанд merge хийгдэхгүй байх тусам conflict үүсэх магадлал нэмэгддэг.

Гэхдээ та deployment environment эсвэл release-д зориулсан тусгай branch-уудыг хадгалахыг хүсэж болно. Жишээ нь, freeCodeCamp repository нь prod-staging болон prod-current branch-уудыг ашиглан staging болон production серверүүд рүү deployment trigger хийдэг. Энэ нь багийг илүү хурдан ажиллах боломж олгодог — бэлэн, approve болсон зүйлсийг merge хийх бүрт deployment хийх шаардлагагүй. Харин deployment нь main branch-ийн хамгийн сүүлийн хувилбарыг эдгээр production branch-ууд руу merge хийх замаар хийгддэг.

Мөн та “release branches” буюу бүтээгдэхүүний томоохон version-ийг илэрхийлэх branch-уудыг хадгалахыг хүсэж болно. Ингэснээр version бүрийн цэвэр төлөвийг хадгалж, өмнөх version-д чухал security patch хийх шаардлагатай бол шинэ version-ийн өөрчлөлтүүдийг оруулахгүйгээр засвар хийх боломжтой болно.

Эцэст нь, branch-уудаа local CLI ашиглан шууд main руу merge хийхээс зайлсхийх хэрэгтэй. Үүний оронд өөрчлөлтүүдээрээ pull request үүсгээд тэр урсгалаар merge хийх нь зөв. Ингэснээр таны CI (continuous integration)-ийн test-үүд амжилттай байгаа эсэхийг шалгах, мөн diff view-ийг дахин шалгаж зөвхөн хүссэн өөрчлөлтүүдээ merge хийж байгаа эсэхийг баталгаажуулах боломжтой болно (та энэ талаар дараагийн хичээлд илүү дэлгэрэнгүй судална).

Эдгээр зөвлөмжүүдийг дагаснаар та branch ашиглан амжилтад хүрэх боломжтой!
