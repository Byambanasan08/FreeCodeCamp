# Merge Conflicts гэж юу вэ, мөн тэдгээрийг хэрхэн шийдвэрлэх вэ?

Хэрвээ та туршлагатай хөгжүүлэгчдийн дунд байсан бол, онлайн community-д эсвэл бодитоор уулзаж байсан ч бай, та “аймшигтай merge conflicts” (dreaded merge conflicts) гэдэг зүйлийн талаар сонсож байсан байх магадлалтай. Гэхдээ тэд яг юу вэ?

За тэгвэл, хэрвээ та өмнөх хичээлүүд дээр ярьсан шиг зөв branching strategy ашиглаж байгаа бол, та нэгэн зэрэг хоёр өөр branch дээр хоёр өөр feature дээр ажиллаж байж болно. Эсвэл та ямар нэг зүйл дээр ажиллаж байхад, танай хамтрагч өөр branch дээр өөр зүйл дээр ажиллаж байж болно. Аль ч тохиолдолд, хоёр бие даасан өөрчлөлт зэрэг явагдаж байна гэсэн үг. Гэхдээ заримдаа, эдгээр хоёр branch хоёулаа нэг файлыг засварлах тохиолдол гардаг. Хэрвээ нөгөө branch эхэлж main руу merge хийгдвэл, таны branch-ийн өөрчлөлтүүд одоо хоцрогдсон (behind) байдалд орно.

Ихэнх тохиолдолд, үүнийг main-ийг буцааж таны feature branch руу merge хийснээр засаж болдог. Гэхдээ нөгөө branch-ийн өөрчлөлтүүд таны branch-ийн өөрчлөлтүүдтэй зөрчилдвөл (conflict), Git merge хийх явцад аль өөрчлөлтийг хадгалах ёстойг тодорхойлж чаддаггүй. Үүнийг “merge conflict” гэж нэрлэдэг.

Тэгвэл үүнийг яаж засах вэ? За тэгвэл, жижиг бөгөөд энгийн зөрчлүүдийн хувьд GitHub танд үүнийг засах web UI өгдөг. Хэрвээ зөрчил (conflicts) байвал PR-ийн доод хэсэгт “Resolve conflicts” товч гарч ирнэ. Та үүнийг дарвал, таны код болон зөрчилдөж буй кодыг агуулсан файлыг харах болно. Git нь таны код болон зөрчилдөж буй кодын эргэн тойронд “merge conflict markers” нэмдэг бөгөөд ингэснээр асуудлыг танихад тусалдаг. Энэ нь дараах байдлаар харагдана:

1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main

Эхлэх marker нь <<<<<<< feat/conflict бөгөөд энэ нь одоогийн branch-ийн хийж буй өөрчлөлтүүдийн эхлэлийг илэрхийлдэг. Дараа нь Here's some code from my first PR :) гэсэн мөрийг харна. Үүний дараа ======= marker гарч ирэх бөгөөд энэ нь манай өөрчлөлтүүдийн төгсгөл болон main branch-ийн өөрчлөлтүүдийн эхлэлийг илэрхийлдэг. Main дээр хийгдсэн өөрчлөлт нь Here's some code that was merged to main that conflicts with the code on my PR гэсэн мөр гэдгийг харж болно. Үүний дараа >>>>>>> main гэсэн хаах marker гарч ирж, зөрчил дууссаныг илэрхийлдэг.

Гэхдээ бид үүнийг яаж засах вэ? За тэгвэл, та merge conflict-ийн нэг хэсэг болгон аль өөрчлөлтүүдийг хадгалахыг шийдэх хэрэгтэй. Хэрвээ та хоёр өөрчлөлтийг хоёуланг нь хадгалахыг хүсвэл, зөвхөн conflict markers-ийг устгаад, өөрчлөлтүүдийг хэвээр нь үлдээж болно. Эсвэл, хэрвээ та зөвхөн нэг branch-ийн өөрчлөлтийг хадгалахыг хүсвэл, conflict markers болон нөгөө branch-ийн (хадгалахыг хүсэхгүй байгаа) өөрчлөлтүүдийг устгана. Эцэст нь, хэрвээ танд аль ч өөрчлөлт таалагдахгүй бол, эхлэх marker-ээс эхлээд хаах marker хүртэлх бүхнийг (inclusive) устгаж болно.

Main branch-ийн өөрчлөлтүүдийг устгаад, харин өөрсдийнхийг үлдээе. Одоо бидэнд байгаа нь:

1 Here's some code from my first PR :)

"Mark as resolved" товчийг дарна уу, энэ нь таны шийдвэрт сэтгэл хангалуун байгааг GitHub-д мэдэгдэнэ. Хэрвээ нэмэлт файлууд байсан бол GitHub дараагийн файл руу шилжих байсан. Гэхдээ энэ нь бидний цорын ганц зөрчилтэй файл тул, та "Commit merge" товч гарч ирэхийг харах болно. Тэр товчийг дарснаар GitHub танд зориулж merge commit үүсгэнэ. Та буцаад pull request руу очих бөгөөд тэнд таны шинэ merge commit болон шийдвэрлэгдсэн зөрчлүүдийг харах болно. Ингээд бид бүгд бэлэн боллоо!

Гэхдээ хэрвээ та GitHub дээр шууд зөрчлүүдийг засаж чадахгүй бол яах вэ, жишээ нь тэд хэтэрхий төвөгтэй байвал?

За тэгвэл, хэрвээ таны branch цөөн commit-тэй эсвэл тухайн файлыг зөвхөн хэдхэн commit өөрчилсөн бол, та main-ийг өөрийн branch руу merge хийж чадна:

git checkout feat/conflict
git fetch origin
git merge origin/main

Git энэ merge-ийг боловсруулах үед, та гараар шийдвэрлэх шаардлагатай зөрчлүүдийг авах болно. Git аль файлуудыг засах шаардлагатайг танд мэдэгдэнэ. Бид зөрчилтэй файлуудыг editor дээрээ нээхэд, өмнө харж байсан шиг conflict markers гарч ирнэ, мөн VSCode-ийн ачаар зарим нэмэлт туслах UI (user interface) харагдана. Одоо файлын зөрчлүүдийг өмнөхтэй адил аргаар засна. Гэхдээ энэ удаа, бид commit-ийг гараар үүсгэх хэрэгтэй болно. Энэ нь бусад commit үүсгэхтэй яг адил:

git add .
git commit -m "chore: resolve conflicts"

Эдгээр командыг ажиллуулсны дараа, Git бидний шийдвэрийг хүлээн авсныг харж болно.

Заримдаа, олон commit эсвэл олон зөрчилтэй үед, rebase ашиглах нь ашигтай байж болно. Өөрчлөлтүүдийг merge хийхийн оронд, rebase нь таны branch дээрх бүх commit-уудыг авч, main дээрх хамгийн сүүлийн өөрчлөлтүүдийн дараа дахин байршуулдаг. Git-ийн нэр томъёогоор, энэ нь таны branch-ийн суурийг main дээрх хамгийн сүүлийн commit руу “reset” хийж байгаа мэт болдог.

Үүнийг туршихын тулд, бидэнд шинэ branch хэрэгтэй:

git checkout main
git pull
git checkout -b feat/rebase

Одоо өөрийн branch дээр олон commit үүсгэе, ингэснээр бид rebase хийхийг дадлагажуулж чадна:

echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"

Одоо main дээр өөр нэг өөрчлөлт хийж, зөрчил үүсгэе:

git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"

Тэгээд өөрийн rebase branch руу буцъя:

git checkout feat/rebase

Дараа нь git rebase -i main командыг ажиллуулна. Энэ нь таны branch-ийг main branch-ийн одоогийн төлөв дээр дахин байршуулна. -i flag нь Git-д interactive mode дээр rebase хийхийг заадаг. Та өөрийн commit-уудыг pick гэсэн үгийн хамт харах болно. pick нь Git-д тухайн commit-ийг ямар ч өөрчлөлтгүйгээр ашиглахыг заадаг. Файлыг хадгалаад хаахад, зөрчил байгааг хэлсэн алдааны мессеж гарч ирнэ.

Манай change one commit нь change four commit-тэй зөрчилдөж байгаа тул бид үүнийг шийдэх хэрэгтэй. Файлыг editor дээрээ нээхэд, өмнө харж байсан шиг UI гарч ирнэ. Бид өөрсдийн change one өөрчлөлтүүдийг үлдээнэ. Дараа нь, git add . командыг ажиллуулж өөрчлөлтүүдийг нэмээд, git rebase --continue командыг ажиллуулж Git-д rebase үргэлжлүүлэхэд бэлэн болсныг мэдэгдэнэ. Эхний commit дээрх зөрчлийг шийдсэн тул, Git үлдсэн commit-уудыг автоматаар rebase хийж чадна.

Одоо, сүүлийн цэвэрлэгээний алхам болгон, бид гурван change commit-ийг нэг commit болгон squash хийе, ингэснээр түүх (history) илүү цэвэрхэн болно. git rebase -i main командыг дахин ажиллуулж editor руу орно. Дараа нь эхний командыг pick хэвээр үлдээгээд, дараагийн хоёрыг squash болгон өөрчилнө. Та дахин ижил зөрчилтэй тулгарч магадгүй. Хэрвээ тийм бол, өмнөхтэй адил аргаар шийднэ. Зөрчлийг шийдэж, rebase-ийг үргэлжлүүлсний дараа (эсвэл хэрвээ зөрчил байгаагүй бол шууд), шинэ commit message editor гарч ирнэ. Энэ message нь таны шинэ commit байх бөгөөд squash хийсэн гурван commit-ийн өөрчлөлтүүдийг агуулна. Үүнийг change three complete гэж нэрлэе. Файлыг хадгалаад хаахад, Git rebase-ийг дуусгана.

Хэрвээ бид одоо commit history-г шалгавал, бидэнд нэг ширхэг change three complete commit байгаа бөгөөд энэ нь change four commit дээр зөв суурилсан байгааг харах болно. Энэ нь та squash хийсэн commit-уудтай rebase-ийг амжилттай хийж дуусгасныг илэрхийлнэ! Одоо танд merge conflict-ийг амжилттай шийдвэрлэхэд хэрэгтэй бүх хэрэгсэл байна!
