# Pull Request гэж юу вэ, мөн үүнийг хэрхэн үүсгэх вэ?

Өмнөх хичээл дээр бид feature branch үүсгэж, тэр branch дээр шинэ файл commit хийж, дараа нь үүнийг remote repository руу push хийсэн. Харин дараагийн алхам маань юу вэ? Бид GitHub дээр pull request үүсгэх хэрэгтэй.

Pull request гэдэг нь үндсэндээ таны branch-ээс target branch руу өөрчлөлтүүдийг татаж оруулах хүсэлт юм. Pull request нь open source төсөлд кодын өөрчлөлт оруулах үед ашиглагддаг урсгал (flow) юм. Энэ арга нь тухайн төслийн maintainers-д таны өөрчлөлтүүдийг шалгах боломж олгодог. Тэд comment үлдээж, асуулт асууж, засвар санал болгож, (найдвартай бол) таны өөрчлөлтийг approve хийж merge хийж чадна!

Одоо дахиад GitHub дээрх өөрийн repository руу орцгооё.

Одоо, та шинэ branch-аа push хийсний дараа “та branch руу push хийлээ” гэсэн гоёмсог banner болон pull request үүсгэх товчийг харах ёстой. Энэ товч нь зарим алхмуудыг алгасаж, процессыг хялбаршуулдаг, гэхдээ бид одоохондоо үүнийг үл тоомсорлоно. Ингэснээр banner харагдахгүй үед ч та бэлтгэлтэй байна.

Repository хуудасны дээд хэсэгт байгаа pull requests tab дээр дарна уу, дараа нь дахин banner-ийг үл тоомсорлоод "New Pull Request" дээр дарна. Ингэснээр таны хүсэлтийг бэлтгэх UI руу орно.

Одоо энд хэд хэдэн нэр томьёо байна, тэдгээрийг тайлбарлая. Нэгдүгээрт, "base" болон "compare" гэсэн drop-down сонголтууд байна. "base" нь таны merge хийх зорилтот branch юм. Бид main руу merge хийх гэж байгаа тул үүнийг өөрчлөх шаардлагагүй.

"compare" нь таны merge хийх гэж буй зүйл юм. Бид feature branch-аа merge хийхийг хүсэж байгаа тул үүнийг feature болгон өөрчлөх ёстой. Үүнийг мөн head branch гэж нэрлэдэг бөгөөд энэ нь таны санаж байх ёстой нэр томьёо юм.

Та үүнийг хийсний дараа UI шинэчлэгдэнэ. Одоо энэ нь feature дээр байгаа боловч main дээр байхгүй commit-уудыг харуулж, яг ямар өөрчлөлтийг merge хийхийг хүсэж байгааг илэрхийлнэ. Мөн “diff” гэж нэрлэгдэх зүйлийг харах бөгөөд энэ нь таны хийсэн бодит өөрчлөлтүүдийг харагдахуйц (visual) байдлаар харуулсан дүрслэл юм.

Бид зөвхөн нэг файл нэмсэн тул diff нь маш цэвэрхэн харагдана. Илүү төвөгтэй өөрчлөлтүүдийн хувьд GitHub нь мөрийн өөрчлөлт (line modifications), нэмэлтүүд (additions), устгалууд (deletions) гэх мэт зүйлсийг тодруулж харуулна. Diff-ийг уншиж ойлгох нь эхэндээ төвөгтэй байж болох ч, бага зэрэг дадлага хийснээр та үүнд нэлээд чадварлаг болно.

Одоохондоо pull request руугаа буцъя. Дээд хэсэгт “compare across forks” гэсэн жижиг холбоос (link) байна. Хэрвээ бид freeCodeCamp repository-ийн fork дээр ажиллаж байсан бол энэ нь PR-аа (pull request) зөвхөн өөрийн fork доторх main branch руу биш, харин анхны freeCodeCamp repository руу чиглүүлэх боломж олгоно.

Бид fork ашиглаагүй байгаа тул үүнийг үл тоомсорлож болно. Гэхдээ энд хийж буй зүйл маань үндсэндээ суралцах болон туршилтын зорилготой гэдгийг ойлгох нь маш чухал. Та зөвхөн дадлагын өөрчлөлтүүд агуулсан PR-ийг бусдын төсөл рүү нээх ёсгүй — энэ нь maintainers-д илүүдэл “noise” үүсгэдэг.

Одоо бид өөрчлөлтүүддээ сэтгэл хангалуун бөгөөд бүх зүйл тохируулагдсан тул "Create Pull Request" товч дээр дарж болно. Ингэснээр шинэ UI руу орно.

Доод хэсэгт (харагдахгүй хэсэгт) өмнө харсан commit жагсаалт болон diff харагдана. PR үүсгэхээс өмнө сүүлийн удаа давхар шалгах нь үргэлж ашигтай, гэхдээ энэ туршилтын хувьд та үргэлжлүүлж болно.

Мөн та хэд хэдэн шинэ input талбаруудыг харах болно. Эхнийх нь таны pull request-ийн гарчиг (title) юм. Хэрвээ PR-д зөвхөн нэг commit байвал GitHub автоматаар тухайн commit дээр үндэслэн гарчиг үүсгэнэ, харин олон commit байвал branch-ийн нэр дээр үндэслэнэ.

Хоёр дахь input нь тайлбар (description) юм. Ихэнх тохиолдолд энэ хоосон байдаггүй. Харин төслийн maintainers урьдчилан тодорхойлсон template өгч, энэ хэсгийг автоматаар бөглөсөн байдаг. Хэрвээ тийм template байвал та тэдний workflow-ийг хүндэтгэж, тухайн template-ийг бүрэн бөглөх хэрэгтэй.

Хоёр тохиолдолд хоёуланд нь та тухайн төслийн contributing documentation-ийг уншиж, PR-ийн гарчиг болон тайлбарыг хэрхэн бичих зааврыг дагаж мөрдөх хэрэгтэй.

Энэ repository нь таны эзэмшилд байгаа тул баруун талд reviewers, assignees, labels, projects, milestones зэрэг тохиргоонуудыг хийх боломжтойг харах болно. Харин бусдын төсөлд хувь нэмэр оруулах үед эдгээрийг засварлах боломжгүй, зөвхөн review хүсэх боломжтой байдаг. Гэхдээ энэ repository таных тул эдгээр боломжуудыг туршиж үзэхийг хүсвэл дурын байдлаар тохируулж болно.

Бүх зүйл бөглөгдөж, тохируулагдсаны дараа "Create pull request" дээр дарна. Эцэст нь, бодит үр дүнг харах болно!

Та бусдын төсөл рүү pull request илгээх үед энэ нь “code review” дамжлагаар ордог. Энэ нь maintainers таны pull request-ийг шалгаж, өөрчлөлт шаардах, асуулт асуух гэх мэт үйл явц юм. Maintainers зөвшөөрсний дараа тэд үүнийг merge хийж болно.

Харин энэ нь таны өөрийн төсөл тул review хүлээх шаардлагагүй. Үүний оронд Commits tab дээр commit history-ийг, Files changed хэсэг дээр өөрчлөлтүүдийг шалгаад, хэрвээ сэтгэл хангалуун байвал merge хийж болно.

Merge хийсний дараа эдгээр өөрчлөлтүүдийг өөрийн local repo руу татах хэрэгтэй.

Terminal руугаа буцаад, одоо ч repository-ийн folder дотор байгаагаа шалгана. Дараа нь git checkout main ашиглан main branch руу шилжинэ.

Дараа нь git pull ашиглан хамгийн сүүлийн өөрчлөлтүүдийг татаж авна.

Одоо git log ашиглан commit history-ийг харвал таны commit харагдана. Мөн “Merge pull request #1” гэсэн мессежийг ч харах болно. Энэ юу вэ?

GitHub дээр pull request merge хийх үед та гурван төрлийн стратеги ашиглаж болно:

Анхдагч (default) нь "Merge" бөгөөд энэ нь head branch-аас бүх commit-уудыг авчирч, дараа нь merge-ийг боловсруулах, бүртгэх зорилгоор нэмэлт commit үүсгэдэг.

Өөр нэг сонголт нь "Squash and Merge" бөгөөд head branch дээрх БҮХ commit-уудыг нэг commit болгон нэгтгээд, merge commit үүсгэхгүйгээр base branch руу шууд нэгтгэдэг.

Сүүлийнх нь "Rebase and Merge" бөгөөд бүх commit-уудыг авч, тэдгээрийг base branch-ийн хамгийн сүүлийн commit-ийн дараа байрлуулах байдлаар “reset” хийж, дараа нь merge commit үүсгэхгүйгээр нэгтгэдэг. Rebasing нь нэлээд төвөгтэй сэдэв тул дараа нь илүү гүнзгий судлах болно, одоохондоо бүрэн ойлгохгүй байсан ч зүгээр.

Open source төслүүд өөрт тохирсон merge стратегийг сонгодог. Гэхдээ эдгээр ялгаатай зан үйлээс шалтгаалан та үргэлж шинэ branch үүсгэж, өөрчлөлтүүдээ commit хийх ёстой. Хэзээ ч шууд default main branch дээр commit хийж болохгүй, эс бөгөөс анхны repository-оос өөрчлөлтүүд татах үед асуудал үүсч, эмх замбараагүй болно.

Ингээд та өөрийн анхны pull request-ийг амжилттай үүсгэж, merge хийлээ!
