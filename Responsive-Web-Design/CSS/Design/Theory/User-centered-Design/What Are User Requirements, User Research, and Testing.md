# User Requirements, User Research, болон Testing гэж юу вэ?

**User research** гэдэг нь таны бүтээгдэхүүнийг ашигладаг хүмүүсийг **системтэйгээр судлах үйл явц** юм. Үүний зорилго нь хэрэглэгчдийн **хэрэгцээ, зан төлөв, мөн тулгарч буй асуудлуудыг (pain points)** хэмжих явдал юм.

User research нь олон янзын хэлбэртэй байдаг. Тэдгээрийн хамгийн түгээмэл хэлбэрүүдийн нэг бол **Net Promoter Score (NPS)** юм. **NPS** нь хэрэглэгчид таны бүтээгдэхүүнийг **найздаа санал болгох магадлал** хэр байгааг хэмждэг. NPS-ийг ихэвчлэн хэрэглэгчийн аяллын тодорхой үе шатанд санал асуулгаар хэмждэг. Жишээлбэл:

* 7 хоногийн дараа
* 30 хоногийн дараа
* 90 хоногийн дараа

NPS нь **0-ээс 10 хүртэлх хэмжүүрээр** хэмжигддэг бөгөөд **9 болон 10** оноо нь таны сайтыг **идэвхтэй сурталчлагч (active promoter)** хэрэглэгчийг илэрхийлдэг.

Өөр нэг судалгааны арга бол **exit interview** юм. Энэ нь хэрэглэгч **захиалгаа цуцлах** эсвэл **акаунтаа устгах** үед үзүүлдэг санал асуулга юм. Энэ санал асуулгын мэдээлэл нь хэрэглэгчид яагаад үйлчилгээ ашиглахаа больж байгааг ойлгоход тусалдаг бөгөөд ингэснээр та эдгээр асуудлыг **шийдвэрлэх боломжтой** болдог.

Нөгөө талаас **user testing** гэдэг нь хэрэглэгчид таны аппликейшнтэй харилцах үед **мэдээлэл цуглуулах практик** юм. Жишээлбэл, видео тоглоом **beta testing** шатанд туршигдах нь user testing-ийн нэг хэлбэр юм. Вэб хөгжүүлэгчийн хувьд танд түгээмэл таарах жишээ бол **A/B testing** юм. **A/B testing** нь шинэ функцийг хэрэглэгчдийн сангаас **санамсаргүй сонгогдсон хэсэгт** нэвтрүүлж туршихыг хэлдэг. Дараа нь **analytics** мэдээллийг ашиглан тухайн функц **ашигтай эсэхийг** тодорхойлдог.

Эцэст нь **user requirements** гэдэг нь таны аппликейшн **дагаж мөрдөх ёстой шаардлага эсвэл хэрэглэгчийн түүхүүд (user stories)** юм. Эдгээр нь хөгжүүлэлтийн процессыг чиглүүлж өгдөг. User requirements нь **user research**, **салбарын стандартууд**, эсвэл **stakeholder-ийн санал** зэргээс тодорхойлогдож болно.

Эдгээр шаардлагууд нь:

* **functional requirements** — аппликейшн **хэрхэн ажиллахыг** тодорхойлдог
* **non-functional requirements** — аппликейшн **ямар байдлаар ажиллах ёстойг** тодорхойлдог

User requirements нь **тогтмол биш** байдаг. **User testing** болон **user research**-ийн мэдээлэл нь шаардлагад нөлөөлж болох бөгөөд таны хэрэглэгчдийн суурь өөрчлөгдөх тусам шаардлагууд ч мөн **өөрчлөгдөнө**.

Эдгээр ойлголтуудын ялгааг ойлгох нь **хамгийн зөв мэдээллийг цуглуулахад** чухал бөгөөд ингэснээр та эцсийн хэрэглэгчдэд **хамгийн сайн туршлагыг** хүргэх боломжтой болно.
